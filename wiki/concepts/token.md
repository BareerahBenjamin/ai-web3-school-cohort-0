# Token

> 模型处理文本的最小单位；决定上下文容量、调用成本与「模型到底看见了什么」。

## 解释

Token 由 tokenizer 切分，不一定等于一个字或一个词。同一页面在字符上很短，若含大量 JSON、地址、代码，token 数可能很高。Context window 的上限以 token 计，因此产品要做裁剪、摘要、检索按需加载，而不是无脑拼接。

理解 token 有助于评估：多轮对话为何突然「忘记」早期约束、为何账单随输入长度陡增、为何要在 system 里重复关键禁令。

## 例子

- **长合约分析**：只注入相关函数片段 + 签名哈希，而非整份 2 万行源码。
- **成本估算**：客服 Bot 对日志做预摘要再进模型，避免每轮重传全文。
- **多语言混合**：中英代码注释混排时 token 切分不规则，应用字符/token 双监控。

## 误区

- **「页面短 = token 少」**：JSON、base64、表格往往更吃 token。
- **「窗口够大就不用管」**：模型未必有效利用远端 token（lost in the middle）。
- **「中文一字一 token」**：取决于具体 tokenizer。

## 使用边界

| 适合 | 不适合 |
|------|--------|
| 预算、裁剪、监控输入规模 | 用 token 数判断语义完整性 |
| 与 RAG 按需检索配合 | 假设「没超限 = 模型读了全部」 |
