# 上下文（Context）

> 模型本次能看见、能使用、能被影响的信息空间；难点在于分层标注来源、时效与可信度，而非单纯「塞满窗口」。

## 解释

LLM 不会自动连接你的数据库、链上 RPC、用户历史与项目文档——它只基于**当前注入的 context** 生成输出。Context Engineering 要回答：什么能进窗口、以什么身份进去、何时刷新、不可信内容如何隔离。

典型分层包括：**指令层**（系统规则、工具禁令）、**任务层**（用户目标、会话参数）、**证据层**（工具返回、链上查询、检索片段）、**记忆层**（跨会话偏好，须可撤销）。模型若把网页、用户愿望、过期文档与系统规则混在同一优先级，就会出现「看起来很懂、事实全错」或 Prompt Injection 越权。

在 AI × Web3 中，context 是连接概率推理与可验证链上状态的枢纽；见 [[concepts/chain-aware-context]]。

## 例子

- **Agent 读链**：上下文里显式包含 `chain_id`、最新 `balance`、工具返回的 `simulation_result`，而非让用户口头描述余额。
- **RAG 片段标注**：检索段落带 `source_url`、`updated_at`、`doc_version`，模型被要求只基于标注片段断言。
- **会话隔离**：新任务清空上一轮的工具结果与草稿交易，避免状态泄漏导致重复提交。

## 误区

- **「窗口越大越好」**：长上下文仍可能抓不住重点；应配合检索、摘要与结构化字段。
- **「把所有 PDF 扔进去」**：低信噪比内容会稀释关键约束；应按结构切块按需取回。
- **「记忆 = 永久授权」**：记住用户常用地址不等于本次允许转账；资产类记忆须绑定当前授权。
- **「用户说的话就是事实」**：用户目标层与证据层必须分开。

## 使用边界

| 适合 | 不适合 |
|------|--------|
| 分层、标注来源与时效 | 把密钥、主私钥、不可撤销权限写进 context |
| 可刷新状态 + 可撤销记忆 | 长期缓存价格、nonce、权限而不重查 |
| 与 RAG、工具结果配合 | 用 context 替代实时链上校验 |
