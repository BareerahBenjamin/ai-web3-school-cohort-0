# 提示词（Prompt）

> 人与模型之间的任务接口：把目标、边界、输出格式与安全规则写成可执行的沟通协议，而不是一句「神奇咒语」。

## 解释

Prompt 在工程上承担**软约束**：说明角色、任务、可用输入、禁止行为，以及不确定时如何拒答或降级。它提高模型按意图输出的概率，但**不能保证**在恶意文档、冲突指令或错误上下文下始终遵守——真正边界须由代码、权限、校验与审计承担。

好的 prompt 把模糊需求变成稳定工作说明：任务是什么、哪些信息可信、输出是散文还是 JSON schema、哪些动作必须标记为「需人工确认」。在 Agent 场景里，还要区分「解释」与「执行」——研究助手可整理资料，但不能假装结论已验证；交易解释器可标风险，不能替用户点确认。

## 例子

- **结构化交易分析**：要求输出 `action`、`risk_level`、`requires_human_approval`、`uncertainties` 字段，便于下游代码拒绝高风险草稿。
- **Few-shot 风格对齐**：放一好一坏两个「如何解释 DeFi 清算」示例，让模型学会列资产变化与不确定点，而非只翻译函数名。
- **分层指令**：系统规则（不可泄露密钥）与用户目标（分析某地址）分块注入，避免用户输入覆盖系统禁令。

## 误区

- **「越长越专业越好」**：冗长 prompt 易自相矛盾；应拆成任务、输入、禁止、输出四段。
- **「写一句不要被注入就够了」**：Prompt Injection 需靠来源标注、工具 allowlist、参数校验与人审，不能仅靠措辞。
- **「prompt 可以替代权限系统」**：写入 DB、发交易、调支付接口必须在代码层二次校验。
- **「示例写一次永久有效」**：协议升级后旧 few-shot 会反向误导，应与 eval 一起维护。

## 使用边界

| 适合 | 不适合 |
|------|--------|
| 定义任务、格式、拒答策略 | 作为唯一安全边界 |
| 与 schema / 规则评分配合 | 单独拦截签名、转账、删数据 |
| 迭代优化且用 golden set 回归 | 无版本管理的巨型「万能 prompt」 |
