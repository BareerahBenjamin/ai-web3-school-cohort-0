## 北极星

做出 **一个可演示的 AI × Web3 MVP**：单一用户价值点 + 清晰 AI/Web3 分工 + **约束层**（权限、限额、失败路径），并能在黑客松提交 demo 与威胁模型摘要。

## 项目选题框架（来自项目实战指南）

### 第一步：用户问题（先写进 `hackathon/idea.md`）

> 用户到底为什么要用这个产品？

必须写清：谁在用、什么场景、现状为何低效、你的方案为何更好。

### 第二步：AI 与 Web3 分工

| 层 | 负责 |
|----|------|
| AI | 理解、生成、推荐、规划 |
| Web3 | 资产、身份、透明记录、激励或治理 |

### 第三步：MVP 三问

1. 最小输入是什么？
2. 最小处理流程是什么？
3. 最小输出是什么？

详见 `hackathon/mvp-阶段.md`。

### 推荐 MVP 形态（你有开发基础，建议顺序）

1. **研究助手型**（低风险，先验证价值）→ Bootcamp 第 2–3 周
2. **监控提醒型**（链上只读 + 告警）→ 第 3 周
3. **半自动执行型**（用户确认后执行）→ 黑客松周

## 赛道选择（与你的目标对齐）

| 优先级 | 赛道 | 理由 |
|--------|------|------|
| 主赛道 | 钱包与权限 | 与 Agent 执行边界、半自动 MVP 直接相关 |
| 副赛道 | 开发工具 | 契合独立开发；可做合约/交易解释工具 |
| 产品文档 | 项目指南 + 案例 | 支撑产品研究线，避免范围膨胀 |

## 架构最小四层（每个实验都要对照）

1. **输入层** — 用户输入、钱包状态、链上数据
2. **处理层** — 模型理解、规则、策略
3. **输出层** — 摘要、建议、交易候选
4. **约束层** — 权限、白名单、金额上限、风险阈值（**必须有**）

## 失败方式清单（项目启动前必填）

在 `hackathon/idea.md` 或 `hackathon/风险与失败路径.md` 中回答：

- [ ] 数据拿不到怎么办？
- [ ] 模型判断失误怎么办？
- [ ] 链上调用失败怎么办？
- [ ] 钱包权限超出预期怎么办？
- [ ] 用户不信任系统怎么办？

## 价值验证标准

用户首次使用后能否明确感到：节省时间、降低风险、信息更清楚、或操作更简单？若不能，继续缩小 MVP。
